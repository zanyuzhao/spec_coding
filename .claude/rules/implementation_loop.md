# 自循环实现（需求分析 → 任务拆分 → 逐项开发与测试）

**前置条件**：仅当用户**已确认**需求说明与设计文档（或已按用户要求修改文档），并明确说「继续开发」「按文档开发」「开始实现」等之后，才执行本流程。若当前 active 仍处于「待用户确认文档」状态（刚产出 spec 未确认），应先提示用户查看并确认文档，再说「继续开发」后再实现。

当用户要求「按某 active spec 实现」「实现当前需求」「从 spec 做到可运行/测试通过」或已确认文档后说「继续开发」时，必须按本流程执行，直到验收通过或明确需要用户决策为止。

---

## 复杂度分级与粒度控制

在任务拆分前，先评估需求复杂度：

| 复杂度 | 定义 | 执行策略 |
|--------|------|----------|
| **简单** | 1-2 个端点 + 基础 UI，无复杂状态 | 一次完成，中间无需汇报 |
| **中等** | 3-5 个端点 + 复杂交互，或涉及状态管理 | 分两轮，中间汇报进度 |
| **复杂** | 多表/多模块联动，或涉及架构变更 | 强制拆分多个 spec 或多轮实现 |

**复杂需求处理**：
- 中等复杂度：完成 50% 时汇报，询问是否继续
- 复杂需求：主动建议拆分，每完成一个子模块汇报一次

## 执行流程

### 1. 读 spec

读取 `docs/spec/active/` 下指定的 spec 文件（或用户指明的 active 项）。若 spec 同目录下有 `tasks.md`，以其中任务列表为主；否则根据 spec 内容自行拆分任务。

### 2. 需求分析与任务拆分

- 从 spec 中提炼：接口/端点、数据模型、前后端改动点、边界与错误处理。
- **评估复杂度**：按上表判断是否需要分轮。
- 输出**可执行任务列表**（顺序可依依赖关系）。
- 若 spec 中已有「任务拆分」或「验收」段，直接采用并补全遗漏项。

### 3. 逐项实现（契约驱动）

按任务列表顺序实现，每完成一类或一项：

1. **先对齐契约**：参考 spec 中的 API 契约、数据模型、字段对照表。
2. **实现代码**：遵守 `docs/spec_process/CHECKLISTS.md` 的「开发清单列表」及本目录下 `fastapi_shield.md`、`frontend_architect.md`、`global_guard.md`。
3. **契约验证**：完成后执行契约校验（见下文）。
4. **小步提交**（可选）：每完成一个子模块可做一次提交。

### 4. 契约校验（每完成一个子模块后执行）

**后端路由审计**：
```bash
# 方式一：查看 OpenAPI 文档
# 打开 http://127.0.0.1:8000/docs，对比与 spec 是否一致

# 方式二：列出所有路由
cd backend && python -c "from app.main import app; print('\n'.join([f'{r.methods} {r.path}' for r in app.routes if hasattr(r, 'methods')]))"
```

**前端调用审计**：
```bash
# 搜索前端 API 调用
grep -rn "fetch\|axios" frontend/src --include="*.ts" --include="*.tsx" | head -20
```

**类型对照检查**：
- 对比 `backend/app/schemas/` 下 Pydantic 模型与 `frontend/src/types/` 下 TS 类型
- 检查字段名、类型、必填性是否与 spec 字段对照表一致

**不一致时**：立即停止，修复后再继续。

### 5. 运行验收

**单元/集成测试**：
```bash
cd backend && pytest tests/test_xxx.py -v
cd frontend && npm run build
```

**API 端点测试**（每个端点完成后）：
```bash
# 启动后端
cd backend && uv run uvicorn app.main:app --reload

# 测试端点（示例）
curl http://127.0.0.1:8000/api/items | jq
```

### 6. 需求完成后的 E2E 验证

**当所有任务完成后，执行完整 E2E 验证**：

1. **启动完整环境**：
   ```bash
   ./docs/plan_auto/init.sh
   ```

2. **浏览器测试**：
   - 打开前端页面 `http://127.0.0.1:3000`
   - 按 spec 用户故事执行关键流程（如：登录 → 列表 → 详情 → 操作）
   - 验证数据正确显示、交互正常

3. **边界条件测试**：
   - 空数据列表
   - 非法输入
   - 分页越界
   - 权限边界

4. **可使用 Playwright MCP 自动化**（可选）：
   ```
   用 Playwright 打开 http://127.0.0.1:3000 并执行 [关键流程]
   ```

### 7. 自循环直到通过或阻塞

- 若验收命令全部通过 + E2E 验证通过 → 视为实现完成，简要回复「实现完成，验收已通过」，并列出已完成的项与验收结果。
- 若某命令失败 → 根据报错修改代码，然后**再次执行验收命令**；重复「修改 → 执行验收」直到全部通过，或遇到必须由用户决定的依赖/环境/产品选择时，明确说明阻塞点并询问用户。

### 8. 不主动归档

本流程只做到「实现完成且验收通过」。归档仅在用户**明确说「归档」「开发完成」**等时，按 `spec_manager.md` 的归档流程执行。

## active spec 建议结构（便于自循环）

为减少反复询问，active spec 建议包含以下结构，便于 Claude 自动拆分与验收：

```markdown
status: proposal | implementation

# 需求简述
（一两句话）

# 任务拆分（可选，有则优先采用）
1. 后端：…
2. 前端：…
3. 验收：…

# 验收
- 后端：`cd backend && pytest path/to/test_xxx.py -v`
- 前端：`cd frontend && npm run build`
（或其它可执行的命令）
```

未写「任务拆分」时，由 Claude 根据 spec 正文与「验收」段自行拆解并执行。

## 用户怎么说会触发本流程

- 「按 docs/spec/active/xxx.md 实现，做到验收通过」
- 「实现当前 active 里的 [需求名]，拆任务、逐项做、跑测试直到过」
- 「从 spec 做到可运行 / 测试通过」
- 「按 [某 active spec] 全流程实现并自测通过」

上述表述均表示：执行「需求分析 → 任务拆分 → 逐项开发与测试 → 运行验收 → E2E 验证 → 失败则修到过」的自循环，无需用户每步确认（除非遇到阻塞或环境问题）。
