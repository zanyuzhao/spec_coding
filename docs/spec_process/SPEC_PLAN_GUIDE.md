# Spec 前 Plan 澄清指引

在创建 `docs/spec/active/xxx.md` 之前，建议先进行一轮「Plan 式澄清」，用多轮问答补齐需求与边界，再落到正式 spec 文档里。这样可以保证：

- 需求范围清晰，避免后期反复推翻；
- 数据库 / 接口 / 交互设计有足够背景信息支撑；
- Spec 文档更聚焦于**达成共识后的决策**，而不是探索过程。

> 注意：本指引只约束「怎么问清楚」，不重新实现 Plan 逻辑。具体提问与规划行为，应尽量使用各自环境的**原生 Plan 功能**（Claude `/plan`），以便自动跟随产品更新。

---

## 1. 何时必须考虑先走 Plan 再写 Spec？

当出现以下任一情况时，建议先做 Plan 澄清，再创建/更新 `docs/spec/active/xxx.md`：

- **需求描述只有一句话或非常粗糙**：如「做个任务看板」「加个报表」「支持导出」等；
- **涉及复杂业务规则或状态机**：如计费、审批流、权限、匹配/推荐、结算；
- **涉及安全、权限、多租户或合规约束**；
- **对性能或规模有要求**：如高并发、大数据量、严格时延目标；
- 你自己感觉「脑子里只有个大概方向，不太敢直接写 Spec」。

如果需求本身已经有详细产品文档、线框图，并且范围很小（例如「在现有页面加一个简单字段展示」），可以直接写 Spec，不强制再跑一轮 Plan。

## 2. Plan 阶段的通用提问框架

无论在 Cursor 还是 Claude 中，Plan 阶段都可以围绕下面几个维度发问并记录结论：

1. **目标与非目标**
   - 这个需求的核心目标是什么？一定要解决什么问题？
   - 明确这次**不解决**什么（非目标），后续超出范围的需求需要新 Spec。

2. **用户与场景**
   - 谁在用？（角色、租户、权限）
   - 在什么场景下使用？（频率、入口、设备形态）

3. **数据与边界条件**
   - 输入/输出分别是什么？有哪些重要字段？
   - 数据量级预估？是否有极端大小/频率场景？

4. **接口与交互方式（粗粒度）**
   - 主要对外接口是页面、API 还是异步任务？
   - 是否需要实时性（WebSocket / SSE）还是可以异步？

5. **安全与权限**
   - 谁能查看/修改/删除？是否有租户隔离要求？
   - 是否涉及敏感数据、审计日志、合规要求？

6. **失败与异常流**
   - 最典型的失败场景有哪些？（例如外部依赖挂掉、用户输入不合法、额度不足等）
   - 遇到失败时对用户和系统各自的期望行为是什么？

7. **验收方式**
   - 如何判断「做完了」？有哪些可执行的验收命令、用例或手工验证路径？

Plan 阶段的产出可以是一份要点列表，在后续写 Spec 时整体吸收进去，而不是另存为正式业务 spec。

## 3. 在 Cursor 中的推荐用法（Agent + Plan 模式）

在 Cursor 环境下，建议按以下方式使用本指引：

1. **用户提出新功能/改需求**，`spec_trigger.mdc` 判断应走 Spec 流程；
2. 若信息不全或边界不清，Agent **通过 `SwitchMode` 切换到 Plan 模式**，并说明：
   - 会先用 Plan 模式把需求范围与边界问清；
   - Plan 只做澄清和决策，不写代码；
   - Plan 结束后会自动回到 Agent 模式并生成 `docs/spec/active/xxx.md`。
3. 在 Plan 模式下，围绕「第 2 节」的提问框架，多轮提问并记录关键信息；
4. 问到足够清晰后，Plan 模式生成一份**精简的决策摘要**（要点列表），交给 Agent 模式；
5. Agent 模式根据决策摘要创建/更新 `docs/spec/active/xxx.md`，内容结构遵循 `SPEC_DOC_TEMPLATE.md`：
   - 需求说明部分引用 Plan 的目标、用户、场景、验收标准；
   - 设计部分引用 Plan 的数据/接口/安全/异常等关键决策；
   - 验收段落写入可执行命令。

这样可以保证：

- Plan 的交互逻辑始终由 Cursor 自身维护（随着版本升级自动增强）；
- Spec 文档只承载「已经决定的内容」；
- Plan 的详细对话留存在会话历史中，需要时可追溯。

## 4. 在 Claude Code 中的推荐用法（原生 /plan）

在 Claude Code 或其他支持原生 Plan 的环境中，推荐的流程类似，只是触发方式不同：

1. 用户提出新功能/改需求；
2. 若判断需求不够清晰，优先使用**原生 Plan 功能**（如 `/plan` 命令或 UI 中的「开始规划」按钮）；
3. 在 Plan 会话中，围绕第 2 节的提问框架，多轮问答并迭代出：
   - 功能范围与优先级；
   - 关键用例与边界条件；
   - 粗粒度的数据、接口、交互思路；
   - 明确的验收方式。
4. Plan 结束后，将**Plan 最终总结中的决策要点**拎出来，整理为：
   - 需求简述 + 背景/目标；
   - 用户/场景 + 核心用例；
   - 接口/数据的粗设计；
   - 验收标准与命令。
5. 再让 Claude 按 `docs/spec_process/SPEC_DOC_TEMPLATE.md`，在 `docs/spec/active/` 下创建或更新对应 spec，将这些决策写进正式文档。

同样地，Plan 对话保留在 Claude 历史中，而 Spec 只存放最终达成的共识。

---

## 5. 与现有 Spec / Plan-Auto 流程的关系

- 本指引**不替代**现有的：
  - `spec_trigger.mdc` 中对「何时走 Spec / 何时走 Plan-Auto」的判定；
  - `spec_implementation_phase` 中的实现自循环；
  - `plan_auto_initializer` / `plan_auto_coding_session` 对 Plan-Auto 模式的实现。
- 本指引只在「**Spec 模式的文档阶段之前**」增加一个推荐步骤：
  - 若需求不清晰 → 先用各自环境的原生 Plan 能力澄清；
  - 之后再按现有的 Spec 流程创建/更新 `docs/spec/active/xxx.md`。

当你在对话中说出类似：

- 「这是一个新功能，需求还比较模糊，先帮我问清楚再写 spec」，
- 「先帮我把范围和边界问全，再生成 active spec」，

Agent 理解后就应优先使用本指引中的 Plan 澄清步骤，再进入正常的 Spec 文档生成与实现流程。
